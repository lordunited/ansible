1. What is BlueStore?

### BlueStore is the default storage backend used by Ceph OSDs. Instead of writing data to an underlying filesystem (like XFS or ext4) and dealing with the associated overhead, BlueStore writes data directly to the raw block device.
2. What Constitutes a "BlueStore Signature"?

### When an OSD is created, BlueStore writes specific identifying metadata and LVM (Logical Volume Manager) tags onto the disk. These signatures include:
    
    OSD Superblock: Contains the OSD UUID, the cluster FSID (File System ID), and the creation time.
    
    BlueStore Metadata: Identifies the partitions (e.g., the block device for data, the shared metadata path, and any separate WAL or DB partitions).
    
    LVM Headers: Metadata that links the device to a Volume Group (VG) and Logical Volume (LV), like ceph-xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx.
    
    These signatures tell the system: "This disk belongs to a specific Ceph OSD."


zap: The action you are performing. It tells the orchestrator to completely wipe, format, and erase all file systems, partition tables,
and BlueStore/LVM metadata from the drive.


How BlueStore Works

### In older versions of Ceph, an engine called FileStore was used. FileStore wrote data to a standard Linux filesystem (like XFS), which created unnecessary overhead.

### BlueStore improves on this architecture by interacting with the disk directly:

    Direct Disk Access: BlueStore consumes the raw block device directly, completely bypassing the host's operating system filesystem overhead.

    RocksDB for Metadata: It uses a high-performance, embedded key-value database called RocksDB to store internal metadata, object locations, and allocation maps.

    Raw Storage for Data: The actual data payload is written directly to the raw partition, resulting in lower latency and significantly higher I/O performance.



The Internal Components of a BlueStore OSD

### When you provision an OSD using BlueStore, the storage device is typically partitioned or mapped into three areas:

    Data Block: The primary, large partition where the actual user data is saved.

    BlueStore DB (Metadata): Houses the RocksDB database, which tracks data placement, allocation, and object indexes (can be placed on a faster NVMe drive for better performance).

    WAL (Write-Ahead Log): A small, high-speed partition used to log incoming writes quickly before they are committed to the data block, ensuring data consistency in the event of power failure.



Internal Partitioning: BlueStore automatically divides the single disk into three hidden partitions:
###
    Block: The main data partition.

    Block.db: The partition for RocksDB metadata.

    Block.wal: The partition for the Write-Ahead Log.



How it looks on your OSD node (e.g., ubuntu-05)

### On your storage node ubuntu-05, Ceph will create two distinct OSD services. Let's look at how the disk space is divided for both drives:
    1. Disk 1: /dev/sda (e.g., OSD ID: 0)

    Data Partition (/var/lib/ceph/osd/ceph-0/block)

    Database Partition (/var/lib/ceph/osd/ceph-0/block.db)

    WAL Partition (/var/lib/ceph/osd/ceph-0/block.wal)

    2. Disk 2: /dev/sdb (e.g., OSD ID: 1)

    Data Partition (/var/lib/ceph/osd/ceph-1/block)

    Database Partition (/var/lib/ceph/osd/ceph-1/block.db)

    WAL Partition (/var/lib/ceph/osd/ceph-1/block.wal)
